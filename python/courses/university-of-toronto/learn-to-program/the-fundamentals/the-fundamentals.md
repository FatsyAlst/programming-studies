### Global Frame vs. Stack Frame in Python

#### Global Frame
- The *global frame* is the top-level environment where your Python program starts running.
- **Global variables** and **functions** (things defined outside any function) live here.
- Exists throughout the whole program—there’s only one global frame.
- Example:
  ```python
  x = 5         # x is global
  def foo():    # foo is a global function
      pass
  ```

#### Stack Frame
- A *stack frame* is created **each time a function is called**.
- Holds that function’s **local variables and parameters**.
- Exists **only while the function is running**; disappears when it returns.
- Stack frames are organized in a "stack":  
  - Each new function call puts a new frame on top.
  - When the function ends, its frame is removed.
- Example:
  ```python
  def bar(y):
      z = y + 2  # z is local to bar, lives only in bar's frame
      return z

  bar(3)        # calling bar creates a new stack frame
  ```

### Summary Table

| Frame type    | Lifetime          | Stores                 | Exists for?      |
|---------------|-------------------|------------------------|------------------|
| Global frame  | Whole program     | Globals, functions     | Entire program   |
| Stack frame   | Per function call | Locals, parameters     | Current call     |

**Tip:**
- When tracing code or debugging, remember:
  - Global variables/functions → Global frame  
  - Function’s locals/params → Stack frame (created new for each call)

When you open a file in Python, it is crucial to handle it correctly and close it afterward. This good practice is important for several reasons:

Opening a file uses a file handle, which is a limited resource managed by your operating system. Each open file consumes one handle, and every system has a cap on how many file handles can be open at a time. If your code leaves files open, you risk reaching this limit, causing failures when opening more files and possibly affecting other programs on your system.

When you write data to a file, Python does not always save it to disk immediately. Instead, it keeps the data in a memory buffer for performance. The actual write to disk may happen later, either when the buffer fills up or when you explicitly close the file. If your file isn’t closed properly, especially if your program crashes first, some or all of your data may be lost.

Keeping files open can cause locking problems, especially on platforms like Windows. An open file may be locked, making it unavailable to other programs until your Python script closes it. This could interfere with backup software, antivirus scans, or other external processes.

The best way to manage files in Python is to use the with statement, also known as a context manager. This automatically closes the file when your work with it is done, even if an error occurs partway through. This reduces the chance of mistakes and ensures that resources are always cleaned up properly.

Example using context manager (recommended):
```
with open("example.txt", "w") as file:
    file.write("Hello, world!")
# File is automatically closed after this block
```
Example without a context manager (not recommended unless you remember to close):

```
file = open("example.txt", "w")
file.write("Hello, world!")
file.close()  # must call close() or resource leak possible
```

Handling files properly by always closing them—ideally with a context manager—protects your data, prevents resource leaks, avoids file locking issues, and ensures your code is robust and maintainable.
